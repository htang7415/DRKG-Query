Implement this project end to end according to `project.md`, using `data/drkg.tsv` as the raw graph.

Requirements:

- Build a clean, reproducible project with one root-level `config.yaml` as the single source of runtime arguments
- Save every generated artifact under `results/`
- Save outputs by step in explicit subdirectories under `results/`, and save the cross-engine comparison only after both baseline engines have finished
- Keep SQL and Cypher semantics matched exactly
- Use the template-mining, selection, benchmarking, timeout, restart, cache-flush, and failure rules from `project.md`
- Make the workflow runnable step by step with `bash ...` commands listed in `command.md`, and also runnable end to end with one command via `bash scripts/run_all.sh --config config.yaml`
- The benchmark harness must programmatically restart PostgreSQL and Neo4j for every query instance
- The harness must attempt OS page-cache flush before each query instance and record when that control is unavailable on this machine
- PostgreSQL benchmarking must run with parallel query disabled, including `max_parallel_workers_per_gather = 0` or an equivalent setting
- Every benchmark Cypher query must prepend `CYPHER runtime=slotted`, and every Neo4j `PROFILE` pass must verify `Runtime SLOTTED`
- All figures must use `fontsize=16`, be saved as PNG with `dpi=600`, and not use titles

Expected repository structure:

- `requirements.txt` for Python dependencies
- `src/` for Python package code
- `scripts/` for runnable entrypoints
- `results/` for all outputs
- `config.yaml` for all paths, DB settings, sampling settings, benchmark settings, and plotting settings

Join-order variants and parameterized benchmark queries must be generated by code rather than stored as static query files.

Required `results/` layout:

- `results/01_setup/`
- `results/02_prepare/`
- `results/03_experiments/`
- `results/04_analysis/`
- `results/05_final/` for the assembled final package: report-ready tables, selected figures, run manifest, and any final summary artifacts

Required workflow:

1. Preprocess `data/drkg.tsv`
   - derive `node_type` from `node_id`
   - drop endpoints of the form `<type>::`
   - deduplicate exact `(src_id, rel_type, dst_id)` triples
   - emit cleaned nodes, cleaned edges, and the Neo4j relation-type mapping
   - use a deterministic Neo4j relation-type sanitization rule that produces legal relationship-type tokens from raw DRKG relation IDs
   - make the mapping collision-safe and reproducible; one acceptable rule is `REL_` + uppercase alphanumeric/underscore sanitization + `__` + first 8 hex chars of SHA1(raw_rel_type)

2. Load both databases
   - PostgreSQL schema: `nodes(node_id, node_type)`, `edges(src_id, dst_id, rel_type)`
   - Neo4j schema: `(:Entity {node_id, node_type})` with directed relationships
   - create the indexes and constraints required by `project.md`
   - capture and log the frozen DBMS memory/cache settings from `project.md`
   - ensure PostgreSQL parallel query is disabled for benchmarking
   - ensure Neo4j benchmark sessions use `CYPHER runtime=slotted`

3. Mine and select templates
   - mine path, triangle, and 4-cycle candidates
   - apply the qualification and ranking rules from `project.md`
   - freeze the final selected templates

4. Analyze the cleaned dataset
   - write dataset-level summary tables under `results/02_prepare/preprocess/`
   - save dataset-level figures under `results/02_prepare/figures/`
   - do this before baseline engine benchmarking so the raw graph is characterized independently of the engines

5. Sample bindings
   - generate hub-anchored and uniform-random anchor sets with a fixed recorded random seed from `config.yaml`
   - use exactly 20 bindings per template-regime for baseline and 5 bindings per template-regime for the join-order study, subject to the shortfall rule in `project.md`
   - save sampled bindings to `results/`

6. Run experiments
   - baseline benchmarking on PostgreSQL
   - baseline benchmarking on Neo4j
   - allow PostgreSQL and Neo4j baseline runs to be launched separately
   - run cross-engine comparison only after both baseline engines have finished and written their outputs
   - PostgreSQL join-order study
   - enforce the timing, restart, OS page-cache, timeout, and logging protocol from `project.md`
   - use exactly 1 warmup execution, 10 measured plain executions, and 1 separate instrumented pass per query instance
   - if a warmup or a plain measured run fails, invalidate the query instance and log failure stage, failure type, timeout value, and completed measured-run count; if only the instrumented pass fails, preserve timing and output metrics, record instrumentation metrics as `NA`, and log the instrumented failure with the same metadata
   - generate join-order SQL programmatically using fully parenthesized left-deep SQL, `join_collapse_limit=1`, `from_collapse_limit=1`, and `CROSS JOIN` or `JOIN ... ON TRUE` for disconnected prefixes, with deferred join predicates placed in the final `WHERE` clause
   - exclude the 2-edge path from the join-order study because it has only one join and no nontrivial order choice
   - classify join orders into connected-prefix and cross-product-inducing groups and save that classification

7. Postprocess outputs
   - aggregate CSV results
   - build the join hypergraph for each selected template
   - classify each selected template as acyclic or cyclic
   - compute the AGM-style bound defined in `project.md` by solving the optimal fractional edge cover problem
   - compare the theory-side quantities against runtime and intermediate-work metrics
   - generate publication-quality tables and figures

8. Assemble final package
   - collect the final report-ready tables, selected figures, run manifest, config snapshot, and summary notes under `results/05_final/`

Implementation rules:

- Every runnable script should accept `--config config.yaml`
- `config.yaml` must include the fixed random seed, DBMS settings, restart/cache-flush controls, warmup/measured/instrumented execution counts, binding counts, timeout values, join-order settings, results paths, and plotting settings
- Create a `scripts/00_setup_env.sh` step that creates a local virtual environment on this machine and installs `requirements.txt`
- Do not scatter arguments across multiple config files
- Store raw logs, intermediate CSVs, selected templates, sampled bindings, benchmark results, theory outputs, comparison outputs, and final figures under the step-specific `results/` subdirectories
- Prefer deterministic file names and stable output formats
- Keep code modular: IO, mining, query generation, benchmarking, theory, comparison, and plotting should be separate modules
- Write concise README-level comments only where needed; do not overcomment
